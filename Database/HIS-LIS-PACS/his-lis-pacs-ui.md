## 1. 整体布局架构

### 1.1 整体结构

采用经典的上下结构 + 左右分栏布局，适配医院日常办公场景，保证操作区域最大化：

```
┌─────────────────────────────────────────────────────┐
│  HIS/LIS/PACS Logo     科室 姓名 角色  通知  退出     │ → Top Navbar (高度固定 64px)
├──────────┬──────────────────────────────────────────┤
│          │                                          │
│  Sidebar │             Main Content                 │
│  (Menu)  │                                          │
│          │                                          │
│          │                                          │
├──────────┴──────────────────────────────────────────┤
│              Footer Copyright & 版本信息             │
└─────────────────────────────────────────────────────┘
```

- **Top Navbar**: 固定顶部，显示系统Logo、当前登录用户信息、通知中心、退出登录等全局操作
- **Sidebar**: 固定左侧，支持折叠/展开，默认宽度 220px，折叠后 64px (仅显示图标)
- **Main Content**: 自适应占据剩余空间，内容超出时滚动，所有业务页面在该区域渲染
- **Footer**: 固定底部，显示版权信息和系统版本号

## 2. 交互规范与UX设计原则

### 2.1 通用交互规范

结合医院工作场景特点，制定以下交互规范：

1. **操作反馈机制**
   - 所有异步操作（保存、提交、删除等）必须给出明确反馈
   - 成功：Toast 显示 "操作成功"，自动关闭
   - 失败：Toast 显示错误原因，保留 5 秒，方便用户阅读
   - 提交中：按钮显示 loading 状态，禁止重复点击

2. **危险操作二次确认**
   - 删除患者信息、删除报告、作废收费、删除影像等危险操作必须弹出二次确认框
   - 确认框明确提示操作后果，例如："此操作将永久删除该患者信息，是否继续？"
   - 确认按钮高亮并标注操作类型（例如 "确认删除"）

3. **加载状态处理**
   - 页面加载：使用骨架屏 (Skeleton) 占位，避免页面跳动
   - 表格加载：显示表格顶部 loading 动画，保持布局稳定
   - 图片/影像加载：显示占位图 + 加载进度

4. **空状态处理**
   - 无数据时显示友好的空状态占位图 + 提示文字
   - 搜索无结果时提示 "未找到匹配数据，请调整搜索条件"
   - 提供操作入口，例如 "新增数据" 按钮

5. **权限控制**
   - 不同科室/角色只能看到对应权限的菜单和按钮
   - 医生只能查看自己患者的信息，护士只能处理负责病区的医嘱
   - 无权限操作按钮置灰或隐藏，接口层面做二次校验

### 2.2 响应式适配

- **大屏幕 (≥1200px)**：侧边栏默认展开，完整显示菜单文字
- **中等屏幕 (768px ~ 1199px)**：侧边栏默认展开，支持手动折叠
- **小屏幕 (<768px)**：侧边栏默认收起，点击顶部汉堡按钮从左侧滑出，遮罩层点击可关闭
- 表格在小屏幕上支持横向滚动

### 2.3 键盘快捷键

为提高医生/护士操作效率，支持常用快捷键：

- `Enter`：触发搜索/提交表单
- `Esc`：关闭弹窗/抽屉/下拉框，取消当前操作
- `Ctrl + S`：保存当前表单/病历/报告
- `Ctrl + F`：聚焦搜索框
- `Alt + 1~9`：快速切换顶部标签页（多标签模式下）

### 2.4 UX设计原则

1. **以医务人员为中心**：操作流程符合医院现有工作习惯，减少学习成本
2. **信息分层展示**：关键信息突出显示，次要信息折叠/收起，避免信息过载
3. **减少点击次数**：常用操作放在显眼位置，支持批量操作
4. **错误预防**：重要操作二次确认，避免误操作导致数据丢失
5. **一致性设计**：相同类型操作保持交互一致，按钮位置、颜色统一
6. **高可用性**：支持长时间操作，页面自动保存草稿，防止浏览器刷新数据丢失

## 3. 色彩规范

针对医疗场景的专业性和安全性，采用以下配色方案：

### 3.1 主色调

| 颜色 | 色值 | 使用场景 | 设计思路 |
|------|------|----------|----------|
| 主蓝 | `#165DFF` | Top Navbar、主按钮、选中状态、链接 | 蓝色代表专业、冷静、可信，符合医疗行业属性，该明度和饱和度不会造成视觉疲劳 |
| 次蓝 | `#E8F3FF` | 背景、hover状态、标记 | 作为主蓝的辅助色，用于区分区域，不会抢主内容的视觉焦点 |

### 3.2 功能色

| 类型 | 色值 | 使用场景 | 设计思路 |
|------|------|----------|----------|
| 成功色 | `#00B42A` | 操作成功、正常状态、已完成 | 绿色在医疗场景代表健康、正常，符合大众认知，明度适中不刺眼 |
| 警告色 | `#FF7D00` | 警告信息、待审核、异常状态 | 橙色比黄色更醒目，适合提醒医务人员关注待处理项，不会像红色那样引起过度紧张 |
| 危险色 | `#F53F3F` | 错误信息、删除操作、异常结果 | 红色作为危险提示，符合通用认知，用于严重错误和危险操作 |
| 信息色 | `#86909C` | 辅助文字、提示信息 | 中性灰色，不干扰主内容，可读性良好 |

### 3.3 中性色

| 类型 | 色值 | 使用场景 |
|------|------|----------|
| 标题文字 | `#1D2129` | 主标题、重要数据 |
| 正文文字 | `#4E5969` | 普通正文内容 |
| 辅助文字 | `#86909C` | 次要信息、提示文字 |
| 边框分割 | `#C9CDD4` | 表格边框、分割线 |
| 页面背景 | `#F2F3F5` | 整体页面背景 |
| 卡片背景 | `#FFFFFF` | 卡片、内容区域背景 |

### 3.4 设计说明

1. **低饱和度配色**：整体配色偏向低饱和度，避免高亮度颜色长时间观看造成视觉疲劳，适合医务人员长时间工作使用
2. **颜色语义一致**：保持颜色含义和大众认知一致，蓝色 = 专业，绿色 = 健康，橙色 = 提醒，红色 = 危险
3. **足够对比度**：文字和背景对比度符合 WCAG 2.0 AA 标准，保证不同视力水平的医务人员都能清晰阅读
4. **状态区分明确**：通过颜色直观区分不同状态（正常/异常/待处理/错误），帮助医务人员快速识别信息

### 4.2.4 交互设计

- **过敏警示**：开立药品医嘱时，自动检查患者过敏史，如果过敏弹出红色醒目警告
- **药物相互作用检查**：自动检查新开医嘱与现有用药的相互作用，如有冲突给出警示
- **医嘱传递**：检验医嘱开立保存后，自动通过 Integration Engine 发送 ORM 消息到 LIS
- **检查医嘱传递**：检查医嘱开立保存后，自动发送 ORM 消息到 PACS
- **医嘱审核**：护士执行前需要审核，医生开立后可以修改/停止
- **批量开立**：支持常用医嘱模板，一键批量开立

- 右上：受邀专家影像浏览
- 左下：文字聊天区域
- 右下：会诊结论输入框
- 顶部工具：同步浏览 / 标注工具

### 6.10.2 交互设计
